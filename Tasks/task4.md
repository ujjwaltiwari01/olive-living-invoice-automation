You are a senior backend architect with 20+ years of experience designing financial automation systems and API integrations. Your task is to analyze the Zoho Books Invoice API schema and build a robust schema normalization and validation layer that ensures all invoices generated by our system strictly comply with Zoho's API requirements.

System Context

We are building a production-grade finance automation system called **Olive Invoice Automation**.

The system pipeline currently looks like this:

Invoice Upload (Streamlit)
→ Image preprocessing (OpenCV)
→ Google Document AI (Invoice Parser)
→ Raw extracted fields
→ Structured JSON
→ Human verification layer
→ Zoho Books API

However, the current structured schema produced by the system is **not fully compliant with the Zoho Books API schema**.

The Zoho API expects a very strict payload format, and the current schema contains:

• incorrect field names
• invalid JSON structure
• UI-oriented field names instead of API fields
• calculated values that Zoho does not allow in request payloads

Your job is to fix this by creating a **Schema Normalization Engine**.

Available Files

You have access to:

1. `invoices.yml`
   This is the official OpenAPI specification for the Zoho Books Invoice API.

2. `sample_invoices(Sample Data).csv`
   This represents real invoice data that will come from OCR extraction.

3. Our current structured schema that looks like this:

{
"Invoice Number": "...",
"Invoice Date": "...",
"Customer Name": "...",
"GST Treatment": "...",
"Currency Code": "...",
"Exchange Rate": "...",
"line_items": [
{
"Item Name": "...",
"Item Desc": "...",
"Quantity": 3,
"Item Price": 996.33
}
]
}

This schema **must be converted into a valid Zoho Books API payload**.

Objectives

You must design and implement a **production-ready schema conversion layer**.

Your solution must include:

1. Schema Mapping Engine
2. Field Validation Layer

4. Human Verification Compatible Output

The system must guarantee that every payload sent to Zoho Books will pass schema validation.

Calculated Field Protection

Zoho automatically calculates these values and they MUST NOT be included in payloads:

sub_total
tax_total
total
balance
payment_made
credits_applied

The system must automatically strip these fields if they appear.

Customer ID Resolver

Zoho requires `customer_id`, not just `customer_name`.

Implement a lookup system:

If customer_id is missing:
→ search Zoho Contacts API using customer_name
→ retrieve customer_id
→ inject into payload

Line Item Validation

Each invoice must contain at least one valid line item.

Validate:

name
rate
quantity

Reject invoices that do not meet these requirements.

Type Enforcement

Implement strict typing:

invoice_number → string
date → YYYY-MM-DD
rate → float
quantity → int
currency_code → ISO currency code

If OCR produces invalid formats, normalize them automatically.

Example:

"15 Feb 2026" → "2026-02-15"

Missing Data Handling

If OCR fails to extract a field:

apply safe defaults.

Examples:

currency_code → "INR"
exchange_rate → 1
item_type → "services"
